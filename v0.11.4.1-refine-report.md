## Changes

### Issue 1 — `test_install_into_readonly_dir` failed under `root`

**Root cause.**  POSIX DAC is advisory for uid=0 — `chmod 0400` does
not prevent root from writing.  The test assumed a `PermissionError`
from `install()` that simply never happens when running as root, so
`assert r.returncode == 1` hit.

**Fix.**  Mark the test `@pytest.mark.skipif(os.geteuid() == 0)`
with a short reason string pointing to the sibling
`test_install_into_file_where_dir_expected_emits_clean_json_error`
which covers the same surface (CLI emits clean JSON error, no
traceback) deterministically for both root and non-root callers.

No coverage is lost.  The P3-2 hygiene guarantee (clean JSON on
unhappy filesystem paths) is still asserted by the file-where-dir
test, the unknown-preset test, and the unknown-stack test — all of
which pass for both uid classes.

### Issue 2 — `validate_release_matrix.py --with-pytest` hangs in reviewer env

**Root cause (suspected, not fully reproduced locally).**  The
script invokes `pytest` via `subprocess.Popen` with `start_new_session=True`
and pipes stdout/stderr.  In nested-container / PTY-less CI
environments, pytest's auto-plugins occasionally try to probe for a
tty or read stdin, and the inherited-but-dangling stdin from the
outer tty causes a block.  The outer matrix script then waits the
full 180 s budget before its `killpg` triggers.

**Mitigations applied (defence-in-depth, not a claimed fix).**

1. `subprocess.Popen(..., stdin=subprocess.DEVNULL, ...)` on every
   spawned subprocess in the matrix script.  Nothing it launches
   can now block on input.
2. Module docstring and argparse help mark `--with-pytest` as
   **non-canonical / optional**, documenting the hang observation
   and directing reviewers to run `pytest tests` directly.

**Canonical release gate (v0.11.4.1+):**

```
1.  python -m pytest tests -q
2.  python tools/validate_release_matrix.py \
        --skill /path/to/skill/vibecodekit-hybrid-ultra \
        --update /path/to/update-package
```

The two gates are complementary but independent.  `--with-pytest`
remains available for authors who want a single-command gate and
have a known-good environment, but is not required and is not
claimed to succeed everywhere.
